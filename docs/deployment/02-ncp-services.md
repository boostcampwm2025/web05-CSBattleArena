# 2. NCP 서비스 가이드

> 비용 최적화 관점에서의 NCP 서비스 선택 및 구성 가이드

## 서비스 선택 기준

### 예산 제약

| 항목 | 값 |
|------|-----|
| **월 예산** | 20~30만원 |
| **핵심 원칙** | NCP Managed Service 최소화, Self-managed 우선 |

### 현재 계획 vs 미래 확장

| 서비스 | 현재 계획 | 미래 확장 시 | 비고 |
|--------|----------|-------------|------|
| **Compute** | Server 2대 (WAS + DB) | + WAS 추가 | 필수 |
| **Database** | Self-managed PostgreSQL | Cloud DB 또는 복제 | 비용 절감 |
| **Load Balancer** | 미사용 | ALB 또는 Network Proxy LB | 다중화 시 |
| **Redis** | 미사용 | Self-managed 또는 Cloud DB | 선택적 |
| **Storage** | Object Storage | + CDN | Frontend 배포 |
| **Monitoring** | NCP 콘솔 기본 | Cloud Insight | 선택적 |

│  ┌─────────────────────────────────────────────────────┐    │
│  │  권장: Standard (2 vCPU, 4GB RAM)                   │    │
│  │  OS: Ubuntu 22.04 LTS                               │    │

│  │  예상 비용: ~6만원/월                                │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
│  DB Server (PostgreSQL)                                     │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  권장: Standard (2 vCPU, 4GB RAM)                   │    │
│  │  OS: Ubuntu 22.04 LTS                               │    │
│  │  Disk: 50GB SSD (데이터 증가 시 확장)               │    │
│  │  예상 비용: ~6만원/월                                │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
│  ⚠️ High Memory 스펙은 비용 대비 효율 낮음 (현재 규모)       │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 서버 타입 비교

| 타입 | 특징 | 권장 용도 | vCPU:Memory 비율 |
|------|------|----------|------------------|
| **Standard** | 범용 워크로드 | 웹 서버, API 서버 | 1:4 |
| **High Memory** | 메모리 집약 | 캐시, 인메모리 DB | 1:8 |
| **High CPU** | CPU 집약 | 연산 작업, 빌드 서버 | 1:2 |

### Server 생성 절차

1. **NCP 콘솔 접속** → Compute → Server
2. **서버 이미지 선택**: Ubuntu 22.04 LTS
3. **스펙 선택**: Standard (2 vCPU, 4GB)
4. **스토리지**: 50GB SSD
5. **Network**: VPC 내 적절한 Subnet 선택
6. **ACG 설정**: 역할별 보안 그룹 할당

│  │                                                      │    │

│  └─────────────────────────────────────────────────────┘    │
│                                                              │
│  ┌─────────────────────────────────────────────────────┐    │

│  ├─────────────────────────────────────────────────────┤    │
│  │  장점:                                              │    │
│  │  • 비용: 서버 비용만 (~6만원/월)                     │    │
│  │  • 완전한 커스터마이징 (pgvector 등)                 │    │

│  │  단점:                                              │    │
│  │  • 수동 백업 설정 필요                               │    │

│  │                                                      │    │
│  │  → 비용 절감을 위해 선택                             │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### Self-managed PostgreSQL 구성

```yaml
# docker-compose.db.yml (DB Server에서 실행)
version: '3.8'

services:
  postgres:
    image: postgres:18
    container_name: cs-arena-postgres
    restart: always
    environment:
      POSTGRES_USER: ${DB_USERNAME}
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: ${DB_DATABASE}
    ports:
      - "5432:5432"  # Private Subnet 내에서만 접근
    volumes:
      - postgres_data:/var/lib/postgresql/data
      - ./backups:/backups
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${DB_USERNAME}"]
      interval: 30s
      timeout: 10s
      retries: 3

volumes:
  postgres_data:
```

### 자동 백업 설정

```bash
# /opt/scripts/backup-postgres.sh
#!/bin/bash

BACKUP_DIR="/backups"
DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="${BACKUP_DIR}/backup_${DATE}.sql.gz"

# 7일 이상 된 백업 삭제
find ${BACKUP_DIR} -name "backup_*.sql.gz" -mtime +7 -delete

# 백업 실행
docker exec cs-arena-postgres pg_dump -U ${DB_USERNAME} ${DB_DATABASE} | gzip > ${BACKUP_FILE}

echo "Backup completed: ${BACKUP_FILE}"
```

```bash
# crontab -e
# 매일 새벽 3시 백업
0 3 * * * /opt/scripts/backup-postgres.sh >> /var/log/backup.log 2>&1
```

## Load Balancer (미래 확장 시)

### WebSocket 지원 확인

**NCP Load Balancer는 WebSocket을 지원합니다.**

참고: [NCP Load Balancer 문서](https://www.ncloud.com/product/networking/loadBalancer)

| LB 유형 | WebSocket 지원 | 용도 | 비고 |
|---------|---------------|------|------|
| **Application LB** | ✅ 지원 | HTTP/HTTPS + WebSocket Upgrade | HTTP 연결 업그레이드로 WebSocket 전환 |
| **Network Proxy LB** | ✅ 지원 | TCP 기반 세션 유지 | 세션 유지 필요한 앱에 적합 |
| **Network LB** | △ (DSR) | 고성능, 대용량 | 세션 유지 제한적 |

### WebSocket 사용 시 핵심 설정

```
┌─────────────────────────────────────────────────────────────┐
│            Load Balancer WebSocket 설정                      │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Application Load Balancer (권장)                           │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  Protocol: HTTPS (443)                              │    │
│  │  Target: HTTP (4000)                                │    │
│  │                                                      │    │
│  │  ⚠️ 핵심 설정:                                       │    │
│  │                                                      │    │
│  │  • Connection Idle Timeout: 3600초 (1시간)          │    │
│  │    - 기본값 60초는 WebSocket에 부족                  │    │
│  │    - 게임 진행 시간 고려하여 충분히 설정             │    │
│  │    - 설정 가능 범위: 60 ~ 3,600초                   │    │
│  │                                                      │    │
│  │  • Sticky Session: Source IP Hash                   │    │
│  │    - 같은 클라이언트가 같은 서버로 연결              │    │
│  │    - WAS 다중화 시 필수                              │    │
│  │                                                      │    │
│  │  • Health Check: GET /api/health                    │    │
│  │    - Interval: 30초                                 │    │
│  │    - Threshold: 3회                                  │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
│  예상 비용: ~3만원/월                                       │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 현재 LB 미사용 이유

| 이유 | 설명 |
|------|------|
| **비용** | 월 ~3만원 추가 (예산의 10~15%) |
| **복잡도** | Redis 없이 Sticky Session만으로 불완전 |
| **현재 규모** | 단일 서버로 충분한 트래픽 |
| **우선순위** | 서버 분리(WAS+DB)가 더 높은 우선순위 |

### LB 도입 시점

- WAS 다중화가 필요할 때 (동시 접속 100명 이상)
- 무중단 배포가 필수일 때
- Redis 도입과 함께 고려

## Redis (선택적)

### Redis 도입의 장점

```
┌─────────────────────────────────────────────────────────────┐
│              Redis 도입 시 얻는 이점                         │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  1. 세션 공유 (WAS 다중화 시 필수)                          │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  현재: In-Memory → 단일 서버에서만 유효              │    │
│  │  Redis: 여러 서버가 동일한 세션 데이터 접근 가능     │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
│  2. Socket.IO Redis Adapter                                 │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  • 서버 간 WebSocket 이벤트 전파                     │    │
│  │  • Player A (Server 1) ↔ Player B (Server 2) 통신   │    │
│  │  • WAS 다중화의 핵심 요소                            │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
│  3. 매칭 큐 원자성                                          │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  • LPUSH/LPOP으로 race condition 방지               │    │
│  │  • 분산 환경에서 안전한 큐 관리                      │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
│  4. 캐싱                                                    │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  • 자주 조회되는 문제 데이터 캐싱                    │    │
│  │  • DB 부하 감소                                      │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
│  5. 서버 재시작 시 데이터 보존                              │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  현재: 서버 재시작 → 진행 중인 게임 소실             │    │
│  │  Redis: 게임 상태가 Redis에 보존되어 복구 가능       │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### Redis 미사용 시 한계

| 한계 | 영향 | 심각도 |
|------|------|--------|
| 수평 확장 불가 | WAS 다중화 시 세션 공유 불가 | 높음 |
| 서버 재시작 시 데이터 소실 | 진행 중인 게임 중단 | 중간 |
| 단일 장애점 | WAS 장애 시 전체 서비스 중단 | 중간 |

### Redis 도입 옵션

```

│                  Redis 도입 옵션                             │

│  └─────────────────────────────────────────────────────┘    │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### Redis 도입 결정 기준

| 조건 | 도입 여부 |
|------|----------|
| 단일 WAS 운영 | 선택적 (게임 상태 보존 원하면 도입) |
| WAS 다중화 필요 | **필수** |
| 서버 재시작 시 게임 유지 필요 | 권장 |
| 예산 여유 없음 | 미도입 (현재 상태 유지) |

## Networking

### VPC 구성

```
┌─────────────────────────────────────────────────────────────┐
│                      VPC 구성                                │
├─────────────────────────────────────────────────────────────┤

│  │  • 외부 인터넷 접근 가능                             │    │

│                                                              │
│  ┌─────────────────────────────────────────────────────┐    │

│  • DB 서버에서 외부 패키지 다운로드 시 필요                 │
│  • 또는 초기 설정 후 제거하여 비용 절감                     │
│  • 비용: ~3만원/월                                          │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### DB 서버 접근 방법

```
┌─────────────────────────────────────────────────────────────┐
│               Private Subnet DB 접근 방법                    │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  방법 1: WAS 서버 경유 (권장 - 추가 비용 없음)              │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  1. SSH로 WAS 서버 접속                             │    │
│  │     $ ssh -i key.pem user@was-public-ip             │    │
│  │                                                      │    │
│  │  2. WAS에서 DB 서버로 SSH                           │    │
│  │     $ ssh user@db-private-ip                        │    │
│  │                                                      │    │
│  │  또는 SSH 터널링 (로컬에서 직접 DB 접근):           │    │
│  │     $ ssh -L 5432:db-private-ip:5432 user@was-ip   │    │
│  │     $ psql -h localhost -U postgres                 │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
│  방법 2: Bastion Host (보안 강화 시)                        │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  • 별도 Bastion 서버 (Micro 스펙)                   │    │
│  │  • 추가 비용: ~2만원/월                              │    │
│  │  • WAS 서버와 역할 분리                              │    │
│  │  • 현재는 비용 대비 불필요                           │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
│  방법 3: SSL VPN (NCP 제공)                                 │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  • VPN으로 VPC 내부 직접 접근                       │    │
│  │  • 추가 비용: ~3만원/월                              │    │
│  │  • 여러 관리자가 접근해야 할 경우 유용              │    │
│  │  • 현재는 비용 대비 불필요                           │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
│  현재 권장: 방법 1 (WAS 경유)                               │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## Container Registry

### NCP Container Registry

Docker 이미지를 저장하고 관리하는 프라이빗 레지스트리입니다.

#### 사용 예시

```bash
# NCP Container Registry 로그인
docker login -u {access_key} -p {secret_key} {registry_endpoint}

# 이미지 태깅
docker tag web05-backend:latest \
  {registry_endpoint}/web05-backend:v1.0.0

# 이미지 푸시
docker push {registry_endpoint}/web05-backend:v1.0.0

# 이미지 풀
docker pull {registry_endpoint}/web05-backend:v1.0.0
```

## 비용 요약

### 현재 계획 (최소 구성)

| 서비스 | 스펙 | 월 비용 |
|--------|------|---------|
| WAS Server | Standard (2vCPU, 4GB) | ~6만원 |
| DB Server | Standard (2vCPU, 4GB) | ~6만원 |
| Object Storage | 10GB | ~1만원 |
| 네트워크 트래픽 | 기본 | ~2만원 |
| **합계** | | **~15만원** |

### 확장 시 추가 비용

| 서비스 | 월 비용 | 도입 시점 |
|--------|---------|----------|
| Redis (DB 서버 내) | +0원 | 게임 상태 보존 필요 시 |
| Redis (별도 서버) | +3~4만원 | 리소스 분리 필요 시 |
| Load Balancer | +3만원 | WAS 다중화 시 |
| 추가 WAS Server | +6만원 | 트래픽 증가 시 |
| CDN | +3~5만원 | 글로벌/커스텀 도메인 시 |
| Cloud Insight | +1~2만원 | 모니터링 강화 시 |
| NAT Gateway | +3만원 | Private Subnet 외부 통신 |

### 예산별 구성 예시

| 예산 | 구성 | 특징 |
|------|------|------|
| **~15만원** | WAS + DB + Storage | 최소 구성, 현재 계획 |
| **~20만원** | + Redis (DB 서버 내) | 게임 상태 보존 |
| **~25만원** | + Cloud Insight | 모니터링 강화 |
| **~30만원** | + 추가 WAS 또는 LB | 확장 시작 |
